# Product Requirements Document — Ralph 2.0

## 1. Общее видение (Vision)

Ralph 2.0 — **автономный AI-оркестратор задач** с веб-дашбордом. Запускает Claude Code (или Gemini CLI) через PTY-терминал, последовательно скармливает задачи из spec-файлов и отслеживает прогресс в реальном времени. Пользователь видит живую консоль AI, управляет выполнением (Play/Pause/Stop) и получает отчёты — всё через браузер.

**Ключевое отличие от v1:** Ralph 2.0 — консолидированный проект. Веб-сервер, overseer и агенты живут в одном репозитории. Overseer написан на Node.js (node-pty), а не Python. Используется JSONL dual-channel протокол вместо stdin-инъекций.

## 2. Целевая аудитория

Разработчик-одиночка (или небольшая команда), использующий AI-ассистентов (Claude Code, Gemini CLI) для автоматизации рутинных задач разработки. Хочет:
- Запускать AI на проект и уйти — Ralph выполнит задачи автономно
- Видеть прогресс в реальном времени через браузер
- Управлять несколькими проектами параллельно
- Иметь историю выполнения и отчёты

## 3. Архитектура

### 3.1. Компоненты

| Компонент | Технология | Файл | Описание |
|-----------|------------|------|----------|
| Web Server + Dashboard | Python 3.11, http.server | `ralph-tracker-web.py` | Монолит ~2300 строк. HTML/CSS/JS встроены в Python-строку. Порт 8767. 8 GET + 23 POST эндпоинтов |
| Overseer | Node.js, node-pty | `ralph-overseer.js` | Управляет PTY-процессом AI CLI. JSONL dual-channel. Аудит спринтов, авто-ответы, git commit |
| Idea Worker | Node.js, node-pty | `ralph-idea.js` | Обработка идей пользователя через Claude PTY. Вызывает /task-architect для генерации задач |
| Agent Profiles | Node.js | `agents.js` | Конфигурация CLI-агентов (Claude, Gemini). Boot sequence, patterns |
| System Tray | Python, pystray | `ralph-tray.py` | Иконка в трее Windows (опционально) |
| Spec Converter | PowerShell | `spec-converter-fixed.ps1` | Генерирует spec-файлы из tasks.md |
| TrayConsole Client | Python | `trayconsole_client.py` | Интеграция с TrayConsole (Named Pipes, heartbeat) |

### 3.2. Потоки данных

```
[Dashboard (Browser)]
     ↕ HTTP API (port 8767)
[Web Server (Python)]
     ↕ subprocess + файлы (status.json, live_console_4.log)
[Overseer (Node.js + node-pty)]
     ↕ PTY stdin/stdout + JSONL
[Claude Code / Gemini CLI]
     ↕ файловая система проекта
[Проект пользователя]
```

### 3.3. Файловый IPC

- `status.json` — текущее состояние overseer (running, paused, heartbeat, задача, прогресс)
- `live_console_4.log` — живой лог PTY-вывода для стриминга в дашборд
- `thinking_status.txt` — статус модели (% контекста, текущая thinking-строка, спиннер)
- `.ralph-stop` — файл-сигнал для остановки (overseer проверяет каждую секунду)
- `.ralph-pause` — файл-сигнал для паузы (overseer паузит PTY-процесс)
- `busy_state.json` — состояние "занятости" сервера (переживает перезапуск)
- `launch_state.json` — состояние запущенных проектных процессов (dev-серверы)
- `results/*.json` — результаты выполнения задач (brief + summary, по одному файлу на задачу)
- `crash.log` — детальный лог ошибок overseer (с ротацией >10MB)
- `tracker_state.json` — ID активного проекта (переживает перезапуск сервера)

## 4. Ключевые функции

### 4.1. Трёхколоночный дашборд
- **Колонка 1 (проекты):** Компактный список проектов с прогресс-барами, статус-индикаторами (зелёный/жёлтый/серый/фиолетовый), кнопками Play/Pause/Stop, выбором модели (Sonnet/Opus)
- **Колонка 2 (задачи):** Спринты с чекбоксами, группировка "В работе" / "Выполнено", добавление подзадач, inline-редактирование описаний
- **Колонка 3 (детали):** Описание задачи + результат выполнения + живая консоль AI
- Все разделители перетаскиваемые (resizable), размеры сохраняются в localStorage
- Контекстное меню (правый клик): старт/стоп, пауза, рестарт, генерация спеков, сброс прогресса, полный сброс, удаление проекта

### 4.2. Управление выполнением
- **Play** — запускает overseer для выбранного проекта с выбранной моделью
- **Pause** — замораживает PTY-процесс AI mid-task (мгновенная пауза через `ptyProcess.pause()`)
- **Stop** — корректная остановка: overseer отправляет Escape + `/exit` Claude Code
- **Action Lock** — блокировка кнопок с индикатором загрузки во время операций
- Heartbeat-мониторинг: если overseer не обновляет heartbeat >15 сек — процесс считается мёртвым
- Авто-рестарт при crash'е (до 5 попыток)

### 4.3. Протокол RALPH_RESULT
Overseer ожидает от агента многострочный маркер после выполнения каждой задачи:
```
RALPH_RESULT
TASK: 1.1
BRIEF: <1 предложение: что сделано с точки зрения пользователя>
SUMMARY: <техническое описание: какие файлы создал/изменил>
STATUS: DONE
RALPH_END
```
**Обработка:**
- Парсер чистит markdown-форматирование и box-drawing символы
- Поддерживает неполные блоки (без RALPH_END, если есть STATUS)
- Отвергает placeholder-тексты ("здесь напиши", "что конкретно сделал")
- Fallback: legacy XML-формат (`<promise>`, `<summary>`, `<report>`)
- При успехе: сохраняет результат в `results/`, отмечает [x] в spec.md И tasks.md, пишет в execution_history.md
- Если маркер не получен — 3 retry-запроса с 5-минутным таймаутом каждый

### 4.4. Автономность агента
- **Одна задача за раз:** overseer назначает ровно одну задачу, ждёт RALPH_RESULT, затем назначает следующую
- **Авто-ответ на вопросы:** детектирует вопросительные паттерны (русские и английские) и отправляет "работай автономно, не задавай вопросов"
- **Rate limit:** авто-ответ не чаще 1 раз в 30 секунд

### 4.5. Аудит спринтов и сброс контекста
После завершения всех задач спринта overseer автоматически:
1. Запускает скилл `/ralph-auditor` для проверки качества
2. Аудитор может вернуть `OK` (спринт закрыт) или `FIX` (добавлены доработки)
3. При `FIX` — доработки добавляются как подзадачи (номера .5, .6, ...), выполняются, затем повторный аудит
4. Максимум 3 попытки аудита на спринт
5. После закрытия спринта — автоматический `git add -A && git commit`
6. **Сбор недостающих результатов:** перед перезапуском overseer проверяет наличие файла результата (`results/{task_id}.json`) для каждой задачи спринта. Для задач без результата — запрашивает отчёт у агента (таймаут 120 сек), пока сессия ещё жива
7. **Перезапуск Claude Code:** после закрытия спринта и сбора всех результатов — автоматический `restartAgent()` с новым `session-id`. Это сбрасывает контекстное окно, и следующий спринт начинается со свежим контекстом. Перезапуск после спринта не считается аварийным (не увеличивает `totalRestarts`)

### 4.6. Живая консоль
- Polling (каждую 1 сек) файлов `live_console_4.log` + `thinking_status.txt`
- Статус модели: спиннер (BOOT/THINK/IDLE), % использования контекста
- Кнопки: копирование (статус / основной вывод), очистка, сохранение в файл
- Сворачивание/разворачивание по клику, вертикальный ресайзер

### 4.7. Мульти-проектность
- Добавление проектов через скилл `/ralph-add-project`
- Каждый проект — отдельная директория с `tasks.md`, `specs/`, `.ralph-runner/`
- Параллельное выполнение: несколько overseer-ов для разных проектов
- Контекстное меню: рестарт, сброс прогресса, генерация спеков, удаление
- Данные проектов хранятся в `projects.json` и `projects/`

### 4.8. Управление задачами в UI
- Inline-редактирование описаний задач
- Добавление подзадач к существующим спринтам (авто-нумерация TASK ID)
- Добавление идей через модальное окно → AI анализирует и создаёт задачи
- Toggle чекбоксов задач (с подтверждением при снятии выполненных)
- Просмотр отчётов выполнения (brief + summary)
- Подсветка новых задач (glow-анимация)

### 4.9. Система идей
- Пользователь вводит идею через UI → попадает в очередь
- Батчинг: несколько идей за короткое время группируются
- Worker-тред запускает `ralph-idea.js` — спавнит Claude PTY
- Claude вызывает `/task-architect` для анализа и генерации задач
- Идеи дополняют существующие tasks.md, не перезаписывают

### 4.10. Launch System
- Запуск dev-серверов проекта через `launch.json`
- **Типы шагов:** `command` (выполнить и ждать), `shell` (открыть терминал), `open` (браузер/файл)
- Multi-step launch (несколько процессов последовательно)
- AI-генерация `launch.json` — Claude анализирует проект и создаёт конфиг
- Авто-детекция: HTML → open in browser, Node.js → npm install/build/start, Python → python main.py
- Уникальный порт для каждого проекта (hash от ID, диапазон 8100-9899)
- Сохранение/восстановление состояния запущенных процессов при перезапуске сервера

### 4.11. Интеграция с TrayConsole
- Named Pipes для heartbeat и команд
- Действие "Открыть дашборд" из трея TrayConsole
- Сигнальный файл `console_signal` для показа окна консоли

## 5. Платформа и ограничения

- **Windows 10+ only** — использует WinAPI через ctypes (OpenProcess, SetConsoleCtrlHandler), taskkill, WMIC
- **`os.kill(pid, 0)` УБИВАЕТ процесс на Windows** — используется `OpenProcess` для проверки
- Монолитный файл `ralph-tracker-web.py` (~2300 строк) — HTML/CSS/JS встроены в Python raw-строку
- Vanilla JS без фреймворков
- UTF-8 кодировка, `projects.json` может иметь BOM (`utf-8-sig`)
- Двойное логирование: консоль + web_server.log (класс Tee)
- Ротация логов overseer (>10MB → обрезка до 1MB хвоста, бэкап сессий до 20 штук)

## 7. Текущее состояние

Проект функционирует и активно используется. Основной функционал работает: мульти-проектный трекер, автономное выполнение задач через overseer, живая консоль, аудит спринтов, система идей.

Известных нерешённых задач нет.

## 8. Метрики успеха

- Все проекты пользователя управляются через единый дашборд
- AI выполняет задачи автономно без ручного вмешательства (одна задача за раз, RALPH_RESULT протокол)
- Спринты проходят аудит качества перед закрытием
- Ошибки overseer логируются и видны в UI (не проглатываются)
- Нет XSS-уязвимостей в динамическом HTML
- Перезапуск сервера не теряет состояние запущенных процессов
- Идеи пользователя автоматически превращаются в структурированные задачи
